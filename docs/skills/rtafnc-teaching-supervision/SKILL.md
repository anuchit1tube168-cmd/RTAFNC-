# RTAFNC Teaching Supervision Skill

## 1. ภารกิจของ Skill

Skill นี้เป็นสมองกลางของโครงการ **“ระบบนิเทศการสอน วิทยาลัยพยาบาลทหารอากาศ กรมแพทย์ทหารอากาศ”** ใช้ควบคุมการวิเคราะห์ ออกแบบ พัฒนา ทดสอบ จัดทำเอกสาร และปรับปรุงระบบให้ทำงานตามกระบวนการนิเทศจริง โดยห้ามลดทอนความถูกต้องเพื่อความสวยงาม และห้ามสร้างข้อมูลทางราชการขึ้นเองเมื่อไม่มีหลักฐานต้นทาง

## 2. คำสั่งเปิดใช้งาน

เมื่อพบคำใดคำหนึ่งใน `activation_triggers` ให้เข้าสู่โหมดนี้ทันที และดำเนินการตามลำดับ:

1. อ่านบริบทโครงการและไฟล์ต้นทางก่อน
2. ระบุขอบเขตการเปลี่ยนแปลงและผลกระทบ
3. แก้ทีละส่วนโดยไม่ทำลายฟังก์ชันเดิม
4. ทดสอบทุก flow ที่เกี่ยวข้อง
5. รายงานสิ่งที่ทำจริง สิ่งที่ยังเป็น mockup และข้อจำกัด
6. อัปเดตเอกสาร SOP/Wiki/Changelog เมื่อพฤติกรรมระบบเปลี่ยน

## 3. ข้อมูลโครงการที่ต้องยึด

- ชื่อระบบทางการ: **ระบบนิเทศการสอน วิทยาลัยพยาบาลทหารอากาศ กรมแพทย์ทหารอากาศ**
- หน่วยงาน: **วิทยาลัยพยาบาลทหารอากาศ กรมแพทย์ทหารอากาศ**
- คำย่อ: **วพอ.พอ.**
- ตราสัญลักษณ์: ใช้ไฟล์ตรา วพอ. ที่ได้รับจากผู้ใช้ ห้ามแทนด้วย icon ทั่วไปในส่วนหัวหลักและรายงานทางการ
- บุคลากรต้นทาง: บัญชีรายชื่อเดือนกรกฎาคม 2569 รวม 103 ราย
- นักเรียนพยาบาลทหารอากาศต้นทาง: ปีการศึกษา 2569 รวม 252 ราย
- คู่มือหลัก: คู่มือนักเรียนพยาบาลทหารอากาศ ปีการศึกษา 2566
- SOP: รหัส `วพอ.พอ.-SOP-EDU-TS-001` ฉบับ 1.0 สถานะร่างเพื่อพิจารณาอนุมัติ

> หากข้อมูลตำแหน่ง วิทยฐานะ หรือสถานะการปฏิบัติงานไม่อยู่ในแหล่งต้นทาง ให้แสดง “ไม่ระบุ/รอตรวจสอบ” แทนการเดา

## 4. เป้าหมายผลิตภัณฑ์

ระบบต้องรองรับกระบวนการครบวงจร:

1. เลือกผู้รับการนิเทศ
2. กำหนดแบบประเมินตามตำแหน่ง/วิทยฐานะ
3. เลือกระดับประเมิน: คู่บัดดี้ / หัวหน้ากลุ่มสาระหรือหัวหน้าหน่วย / ผู้บริหาร
4. บันทึกข้อมูลผู้ประเมิน
5. ประเมิน PA4 ด้านที่ 1 และด้านที่ 2 รวม 12 ตัวชี้วัด
6. แนบหลักฐานภาพไม่เกิน 2 รูปต่อการประเมิน
7. บันทึกร่างหรือส่งผล
8. ตรวจสถานะครบ 3 ระดับแบบ 0/3 ถึง 3/3
9. สรุปคะแนน จุดเด่น และจุดที่ควรพัฒนา
10. พิมพ์และจัดเก็บรายงานรายบุคคล/ภาพรวม

## 5. หลักปฏิบัติที่ห้ามฝ่าฝืน

### 5.1 ความถูกต้องของข้อมูล

- ห้ามสร้างชื่อบุคลากร ตำแหน่ง วิทยฐานะ คะแนน หรือหลักฐานที่ไม่มีต้นทาง
- แยกข้อมูลต้นฉบับออกจากข้อมูลที่ระบบคำนวณ
- ห้ามแก้คะแนนภายนอกระบบหลังพิมพ์รายงาน
- แสดงแหล่งข้อมูลและวันที่ข้อมูลอย่างชัดเจน
- ใช้เลขประจำตัวหรือรหัสภายในเป็น primary key ไม่ใช้ชื่อเป็น key หลัก

### 5.2 คะแนนและสถานะ

- คะแนนรายตัวชี้วัดอยู่ในช่วง 1–5 เท่านั้น
- คำนวณค่าเฉลี่ยจากรายการสถานะ “ส่งผลแล้ว” เท่านั้น
- ค่าที่แสดงทางการใช้ทศนิยม 2 ตำแหน่งและหลัก `ROUND(value, 2)`
- ห้ามนับร่างเป็นผลสมบูรณ์
- ครบ 3 ระดับเมื่อมีผลส่งแล้วครบ buddy, head, executive
- เกณฑ์ผ่านต้องกำหนดจากคำสั่ง/แบบประเมินที่อนุมัติล่าสุด หากยังไม่มีให้ตั้งเป็นค่า config และติดป้าย “รอรับรองเกณฑ์”

### 5.3 หลักฐาน

- รองรับ JPG, PNG, WebP
- ไม่เกิน 5 MB ต่อไฟล์ เว้นแต่ผู้ดูแลกำหนดใหม่
- ไม่เกิน 2 รูปต่อระดับการประเมิน
- ต้อง preview และลบก่อนส่งได้
- ไม่เก็บข้อมูลผู้ป่วยหรือข้อมูลอ่อนไหวที่ไม่เกี่ยวข้อง
- ในระบบจริงต้องเก็บ Drive file ID, owner, createdAt, mimeType และ hash/ขนาดไฟล์

### 5.4 สิทธิ์และการตรวจสอบย้อนหลัง

- ใช้ Least Privilege
- บัญชีเป็นรายบุคคล ห้ามใช้บัญชีร่วมกัน
- ผู้ประเมินห้ามประเมินตนเอง
- การแก้ผลหลังส่งต้องมีเหตุผล ผู้อนุมัติ และ Audit Log
- ผู้ตรวจสอบอ่านได้แต่แก้ไม่ได้
- ห้ามลบข้อมูลถาวรจากหน้าปกติ ใช้สถานะยกเลิกพร้อมเหตุผล

## 6. บทบาทผู้ใช้

| บทบาท | สิทธิ์หลัก | ข้อจำกัด |
|---|---|---|
| ผู้ประเมิน | สร้าง/แก้ร่าง/ส่งรายการของตน | ห้ามแก้ของผู้อื่นหรือประเมินตนเอง |
| หัวหน้ากลุ่มสาระ/หัวหน้าหน่วย | ติดตามหน่วย ประเมินระดับหัวหน้า ตรวจความครบถ้วน | เห็นเฉพาะขอบเขตที่รับผิดชอบ |
| ผู้บริหาร | ประเมินระดับผู้บริหาร รับรองและดูภาพรวม | ห้ามแก้คะแนนต้นทางโดยตรง |
| ผู้ดูแลระบบ | จัดการข้อมูลหลัก สิทธิ์ แบบประเมิน สำรองข้อมูล | ไม่ควรใช้บัญชี admin ทำงานแทนผู้ประเมิน |
| ผู้ตรวจสอบ | อ่านรายงานและ Audit Log | ห้ามแก้ข้อมูลประเมิน |

## 8. UX/UI Standard

### 8.1 Branding

- ใช้ตรา วพอ. ใน Sidebar, Dashboard header, Login และหัวรายงาน
- หัวเรื่องหลักต้องใช้ชื่อเต็มของระบบ
- สีหลัก: น้ำเงินกองทัพอากาศ / ฟ้า / ขาว พร้อมสีสถานะที่อ่านง่าย
- เว็บใช้ `Noto Sans Thai`; รายงานราชการใช้ `TH Sarabun New/PSK 16 pt` เมื่อสภาพแวดล้อมรองรับ

### 8.2 Status Color

- เขียว = ส่งผลแล้ว/ครบ
- เหลือง = ร่าง
- แดง = ยังไม่ประเมิน/ผิดพลาด
- ส้ม = ต้องแก้ไข/ส่งคืน
- เทา = ไม่อยู่ในเงื่อนไข

### 8.3 Responsive

- Desktop: Sidebar + content
- Tablet: Sidebar ย่อหรือ drawer
- Mobile: Drawer, card layout, ปุ่มหลักเต็มความกว้าง
- ห้ามซ่อนข้อมูลสำคัญด้วย horizontal scroll โดยไม่มี sticky column/header

### 8.4 Accessibility

- Contrast ผ่าน WCAG AA สำหรับข้อความหลัก
- Form มี label จริง ไม่ใช้ placeholder แทน label
- Keyboard navigation และ focus state ต้องเห็นชัด
- รูปมี alt text
- Error ระบุทั้งสี ไอคอน และข้อความ

## 10. สูตรคำนวณ

```text
indicator_percent = ROUND((score / max_score) * 100, 2)
section_average   = ROUND(SUM(section_scores) / COUNT(section_scores), 2)
evaluation_avg    = ROUND(SUM(all_scores) / COUNT(all_scores), 2)
person_average    = ROUND(SUM(submitted_evaluation_avg) / COUNT(submitted_levels), 2)
completion_rate   = ROUND(completed_people / target_people * 100, 2)
```

ต้องป้องกัน division by zero และแสดง `—` แทน 0 ที่ไม่มีข้อมูลจริง

## 11. Validation Rules

- ผู้รับการนิเทศและผู้ประเมินต้องไม่ใช่คนเดียวกัน
- ต้องเลือกรอบประเมิน ระดับ วันที่ และแบบประเมิน
- คะแนนต้องครบทุกข้อก่อนส่ง
- วันที่สิ้นสุดต้องไม่ก่อนวันที่เริ่มต้น
- หนึ่งรอบ + หนึ่งบุคลากร + หนึ่งระดับ ต้องไม่เกิดรายการส่งผลซ้ำ
- รูปเกินจำนวน/ชนิด/ขนาดต้องถูกปฏิเสธก่อนอัปโหลด
- การส่งผลต้องผ่าน server-side validation ในระบบจริง ไม่พึ่ง browser อย่างเดียว

## 12. Reporting Standard

### รายบุคคล

- ตรา วพอ.
- ชื่อเต็มหน่วยงานและชื่อระบบ
- ข้อมูลผู้รับการนิเทศ
- ตารางผล 3 ระดับ
- คะแนนด้าน 1/2 และเฉลี่ย
- รูปหลักฐาน
- จุดเด่น จุดพัฒนา ข้อเสนอแนะ
- ช่องลงชื่อและวันที่

### ภาพรวม

- จำนวนเป้าหมาย/ประเมินครบ/ค้าง
- คะแนนเฉลี่ยตามหน่วย/ประเภท/ระดับ
- รายการที่ยังไม่ครบ
- สรุปประเด็นพัฒนาที่พบซ้ำ
- วันที่และแหล่งข้อมูล

### Print CSS

- A4 portrait เป็นค่าเริ่มต้น
- `break-inside: avoid` สำหรับแถว ตาราง รูป และช่องลงชื่อ
- ห้ามเกิดหน้าว่าง
- หัวตารางต้องซ้ำเมื่อขึ้นหน้าใหม่
- รูปต้องไม่ยืดผิดสัดส่วน

## 14. Workflow ของ Agent

### READ

### BUILD

- ห้าม hardcode รายชื่อ/คะแนนใหม่โดยไม่จำเป็น

### VERIFY

- ทดสอบ navigation, validation, calculations, permissions, persistence, print และ mobile
- เปรียบเทียบผลก่อน/หลัง
- ตรวจ console/build error

### DOCUMENT

- อัปเดต CHANGELOG
- อัปเดต SOP/Wiki เมื่อ workflow เปลี่ยน
- ระบุ mockup กับ production ให้ชัด

## 15. Loop Engineering และหลักฐานการทำงาน

งานวนซ้ำต้องมีองค์ประกอบครบ:

- Owner
- Trigger/Cadence
- Input source
- Stop condition
- Human approval gate สำหรับงานเสี่ยง
- Run log
- Evidence/Receipt

กติกา: **No run log = failed loop; No receipt = failed loop; No owner = failed loop; No stop condition = failed loop.** ชื่องาน loop ให้ลงท้ายด้วย `-loop`

## 16. Definition of Done

งานถือว่าเสร็จเมื่อ:

- ฟังก์ชันตามคำขอทำงานจริง
- ไม่ทำลาย flow เดิม
- คะแนนและสถานะคำนวณถูกต้องจากข้อมูลจริง
- ไม่มี error สำคัญใน console/build
- ทดสอบ desktop และ mobile
- รายงาน A4 ไม่ตัดตาราง/รูป/ลายเซ็น
- Audit/สิทธิ์ที่เกี่ยวข้องถูกตรวจ
- เอกสารและ Changelog ถูกอัปเดต
- ผู้ใช้ได้รับไฟล์หรือ URL ที่เปิดได้จริง

## 17. Stop Conditions

หยุดและขอข้อมูลเพิ่มเติมเมื่อ:

- ไม่มีแบบ PA4 ฉบับอนุมัติแต่ขอให้กำหนดเกณฑ์ทางการ
- ไม่มีวิทยฐานะต้นทางแต่ขอให้จัดกลุ่มตามวิทยฐานะ
- ต้องแก้ข้อมูลบุคคลจำนวนมากโดยไม่มีไฟล์ต้นฉบับ
- ขอเผยแพร่ข้อมูลส่วนบุคคลสู่สาธารณะ
- การเปลี่ยนแปลงอาจลบหรือเขียนทับข้อมูลจริงโดยไม่มี backup

## 18. Output Contract

ทุกครั้งที่ส่งมอบ ให้สรุป:

1. สิ่งที่แก้จริง
2. ไฟล์/ระบบที่เปลี่ยน
3. ผลการทดสอบ
4. ข้อจำกัดหรือส่วนที่ยังเป็น mockup
5. วิธีเปิด/ใช้งาน
6. งานถัดไปที่จำเป็นเพียงหนึ่งรายการ
